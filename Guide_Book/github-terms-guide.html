📚 GitHub 기본 용어

GitHub의 기초가 되는 핵심 개념들

🏠 Repository (레포지토리)
프로젝트를 담는 저장소 - 마치 프로젝트의 집 같은 곳이에요!
🏠 집 = Repository ├── 📂 방 = Directory/Folder ├── 📄 가구 = Files └── 📋 일기장 = Commit History
예시: HumanProject_1이 하나의 Repository입니다

Repository는 GitHub의 가장 기본 단위입니다. 모든 파일, 폴더, 그리고 프로젝트의 변경 히스토리가 저장되는 곳이에요.

  • Public: 누구나 볼 수 있음
  • Private: 팀원만 볼 수 있음
  • Fork: 다른 사람의 레포를 복사
🌿 Branch (브랜치)
나무에서 가지가 뻗어나가듯이, 메인 코드에서 새로운 작업을 위해 분리된 공간
🌳 나무줄기(main) ├── 🌿 가지1 (develop) - 개발 중인 코드 ├── 🌿 가지2 (feature/login) - 로그인 기능 └── 🌿 가지3 (feature/signup) - 회원가입 기능
브랜치 종류:
• main: 최종 완성된 코드 (배포용)
• develop: 개발 중인 코드 (통합 테스트)
• feature/기능명: 개별 기능 개발
💾 Commit (커밋)
변경사항을 저장하는 "저장 버튼" - 게임에서 세이브하는 것과 같아요!
📸 사진첩 = Commit History ├── 📷 2025.8.21 "로그인 완성" ├── 📷 2025.8.20 "회원가입 추가" └── 📷 2025.8.19 "프로젝트 시작"
좋은 커밋 메시지 예시:
✅ "로그인 기능 완성"
✅ "회원가입 버그 수정"
❌ "수정함"
❌ "아아아아"
⬆️ Push (푸시)
내 컴퓨터의 작업을 GitHub 서버에 업로드하는 것
🏠 내 컴퓨터 ➡️ ☁️ GitHub 서버
VS Code에서: "⬆️ Push" 버튼 클릭!
⬇️ Pull (풀)
GitHub 서버의 최신 내용을 내 컴퓨터로 다운로드하는 것
☁️ GitHub 서버 ➡️ 🏠 내 컴퓨터
⚠️ 중요: 작업 시작 전에 항상 Pull을 먼저 해야 해요!
다른 팀원이 수정한 내용을 받지 않으면 충돌이 날 수 있어요.

🔄 워크플로우 용어

팀 협업에서 자주 사용하는 작업 흐름 관련 용어들

🙏 Pull Request (풀 리퀘스트)
내 작업을 팀 코드에 합쳐달라고 요청하는 것 - 줄여서 PR이라고 해요!
👤 개발자: "내가 만든 기능을 메인에 합쳐주세요!" 👥 팀장: "검토 후 승인할게요!"
1

작업 완료

내 브랜치에서 기능 개발 완료

2

PR 생성

GitHub에서 "New Pull Request" 클릭

3

팀장 검토

코드 리뷰 및 승인 대기

4

머지 완료

메인 브랜치에 통합 완료

🔗 Merge (머지)
서로 다른 브랜치를 합치는 과정 - 여러 사람의 작업을 하나로 모으기
🌿 내 브랜치 + 🌳 메인 브랜치 = 🌳 업데이트된 메인
머지 후 결과:
모든 팀원의 작업이 하나의 완성된 프로젝트가 됩니다!
⚡ Conflict (충돌)
같은 부분을 다르게 수정해서 발생하는 문제 - 당황하지 말고 차근차근 해결해요!
😵 A가 수정: login.html 10줄 😵 B가 수정: login.html 10줄 (다른 내용) 🤝 해결: 둘 중 어느 걸 선택할까?
충돌 해결 방법:
1. 팀장이나 해당자들과 상의
2. 어떤 코드를 남길지 결정
3. 수동으로 코드 수정
4. 다시 커밋하기
📝 Issue (이슈)
버그 신고, 기능 요청, 질문 등을 관리하는 게시판 같은 곳
🐛 Bug: "로그인 안됨" ✨ Feature: "검색 기능 추가 요청" ❓ Question: "이 코드 어떻게 작동하나요?"
이슈 작성 팁:
• 제목은 간단명료하게
• 스크린샷 첨부하면 더 좋아요
• 재현 방법을 자세히 설명
🏷️ Tag (태그)
특정 시점의 코드에 버전명을 붙이는 것 - 중요한 순간을 기록해요!
🏷️ v1.0 "첫 번째 완성버전" 🏷️ v1.1 "버그 수정 버전" 🏷️ v2.0 "대규모 업데이트"
언제 사용하나요?
• 프로젝트 1차 완성 시
• 발표용 버전 완성 시
• 중요한 마일스톤 달성 시

🤔 헷갈리기 쉬운 용어 구분

자주 혼동되는 GitHub 용어들을 명확하게 구분해보세요

⬇️ Pull (풀)

의미: 다운로드

행동: GitHub → 내 컴퓨터

예시: "최신 코드를 Pull 받았어요"

🙏 Pull Request (풀 리퀘스트)

의미: 합쳐달라는 요청

행동: 내 작업 → 팀 코드로 합치기

예시: "PR 올렸어요!"

💾 Commit (커밋)

위치: 내 컴퓨터에 저장

행동: 변경사항 기록

예시: "로그인 기능 완성"

⬆️ Push (푸시)

위치: GitHub에 업로드

행동: 커밋을 서버로 전송

예시: "커밋 후 Push까지 완료"

🌿 Branch (브랜치)

범위: 같은 레포지토리 내

용도: 기능별 작업 분리

권한: 팀원 모두 접근 가능

🍴 Fork (포크)

범위: 다른 레포지토리로 복사

용도: 남의 프로젝트 복제

권한: 내 계정으로 독립적 관리

🎯 용어 퀴즈

GitHub 용어를 얼마나 이해했는지 확인해보세요!

💬 팀 협업에서 자주 쓰는 표현

학원 프로젝트에서 들을 수 있는 GitHub 관련 대화들

🗣️ VS Code에서 보는 용어들
Source Control: Git 관리 패널
Initialize Repository: Git 시작하기
Stage Changes: 커밋할 파일 선택
Unstage: 커밋에서 제외
Publish branch: 새 브랜치를 GitHub에 올리기
💭 팀원들이 자주 하는 말
👨‍💻 "PR 올렸어요!" = Pull Request 만들었어요 👩‍💻 "머지해주세요!" = 제 작업을 메인에 합쳐주세요 🧑‍💻 "풀 받으세요!" = 최신 코드를 다운받으세요 👨‍💻 "충돌났어요!" = 같은 파일을 동시에 수정해서 문제생겼어요 👩‍💻 "커밋 메시지 뭐로 할까요?" = 저장할 때 설명 뭐라고 쓸까요?
⚡ 급할 때 쓰는 GitHub 단축 표현
"푸시했어요!" = GitHub에 업로드 완료
"풀하고 시작할게요" = 최신 코드 받고 작업 시작
"브랜치 만들었어요" = 새로운 작업 공간 생성
"커밋만 하고 갈게요" = 저장만 하고 집에 갈게요
"이슈 올릴게요" = 버그나 문제 신고할게요
⚠️ 팀 협업 주의사항
1. 작업 전 항상 Pull 먼저!
2. 커밋 메시지는 명확하게!
3. 큰 변경사항은 팀장과 상의!
4. 충돌 발생 시 혼자 해결하지 말고 도움 요청!
