Entity Relationship Diagram & Database Design

📊 개요

📋 비즈니스 요구사항

  • 사용자들이 회원가입 후 도서에 대한 리뷰를 작성할 수 있어야 함
  • 도서는 제목, 저자, 출판사, 장르, 출간일 등의 정보를 포함해야 함
  • 리뷰는 별점(1-5), 내용, 추천여부를 포함해야 함
  • 한 사용자는 하나의 도서에 하나의 리뷰만 작성 가능
  • 도서별 평균 평점과 리뷰 수를 계산할 수 있어야 함

BookReview 시스템은 사용자(Users), 도서(Books), 리뷰(Reviews) 세 개의 주요 엔터티로 구성됩니다. 이들 간의 관계는 다대다 관계를 리뷰 테이블을 통해 일대다 관계로 해결하는 구조입니다.

🔗 테이블 관계

관계 부모 테이블 자식 테이블 카디널리티 관계 설명
User → Books Users Books 1 : N 한 사용자는 여러 도서를 추가할 수 있음
User → Reviews Users Reviews 1 : N 한 사용자는 여러 리뷰를 작성할 수 있음
Books → Reviews Books Reviews 1 : N 한 도서는 여러 리뷰를 가질 수 있음

📝 제약조건 및 인덱스

🔐 Primary Keys

  • ✅ Users.id - UUID 형태의 고유 사용자 식별자
  • ✅ Books.id - UUID 형태의 고유 도서 식별자
  • ✅ Reviews.id - UUID 형태의 고유 리뷰 식별자

🔗 Foreign Keys

  • 🔸 Books.addedBy → Users.id (도서 추가자)
  • 🔸 Reviews.userId → Users.id (리뷰 작성자)
  • 🔸 Reviews.bookId → Books.id (리뷰 대상 도서)

🚫 Unique Constraints

  • ⚡ Users.email - 이메일 중복 방지
  • ⚡ Reviews(userId, bookId) - 한 사용자당 도서 1개 리뷰

✅ Check Constraints

  • 📊 Reviews.rating BETWEEN 1 AND 5
  • 📏 Users.username LENGTH >= 3
  • 📧 Users.email LIKE '%@%'

⚡ Indexes

  • 🔍 idx_books_title - 도서 제목 검색 최적화
  • 🔍 idx_books_author - 저자별 검색 최적화
  • 🔍 idx_books_genre - 장르별 필터링 최적화
  • 🔍 idx_reviews_book - 도서별 리뷰 조회 최적화
  • 🔍 idx_reviews_user - 사용자별 리뷰 조회 최적화

💡 설계 결정사항

✅ UUID 사용

모든 Primary Key에 UUID를 사용하여 분산 환경에서의 고유성을 보장하고 보안을 강화했습니다.

📊 정규화

3NF(제3정규형)을 준수하여 데이터 중복을 최소화하고 데이터 무결성을 보장했습니다.

⚡ 성능 고려

자주 검색되는 컬럼(title, author, genre)에 인덱스를 추가하여 쿼리 성능을 최적화했습니다.

🔒 데이터 무결성

적절한 제약조건을 통해 잘못된 데이터 입력을 방지하고 비즈니스 규칙을 강제했습니다.
